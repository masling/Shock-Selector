# 产品在线选型工具 MVP - PRD

## 1. 项目背景

当前产品选型逻辑保存在本地 Excel 中，包含 19 个工况 sheet 与 1 个产品数据库 sheet。用户输入工况参数后，Excel 先计算所需能量、行程与推力，再从产品数据库中过滤出满足条件的型号。

现阶段的问题：

- 19 个工况直接暴露给用户，理解门槛高。
- Excel 适合内部使用，不适合对外网站访问、维护与迭代。
- 采购人员和非专业用户很难快速判断该选哪个工况。
- 工况公式、产品库、筛选逻辑混在一起，不利于系统化开发。

本项目目标是将 Excel 选型程序整理为一个可在线使用的站点功能页面，面向两类核心用户：

- 工程师：需要输入更完整参数，查看计算过程与筛选依据。
- 采购人员：希望尽量少填参数，快速得到可采购型号。

---

## 2. MVP 目标

MVP 只解决最核心的问题：

1. 把 Excel 的“数据库”sheet 导入为在线产品库。
2. 把 19 个底层工况整理为更易懂的前台分类。
3. 搭建统一的工况注册表与计算引擎接口。
4. 支持两种入口：工程师选型、采购快速筛选。
5. 先实现 3 个示例工况 calculator，验证架构可行。
6. 输出可扩展的数据结构，使后续 19 个工况可逐步迁移。

MVP 阶段虽然允许按优先级逐步迁移工况，但已落地的 calculator 必须遵守以下原则：

- 输入条件必须与原 Excel 对应工况 sheet 保持一致，不允许为了“更容易填写”而擅自简化字段。
- 如果 Excel 使用了重量、扭矩、安装半径、旋转半径、角速度等原始条件，则前端表单也必须暴露这些原始条件。
- 不允许把 Excel 中的原始条件预先压缩为“转动惯量”等二次推导参数后再让用户填写，除非该原始工况 sheet 本身就是这样定义的。
- 结果区必须展示原公式链路中的关键计算结果值，至少包含动能、做功能量、总能量（每次）、总能量（每小时）、推进力，以及存在时的减速加速度、冲击速度等中间结果，以便用户核对计算正确性。
- 推荐产品筛选可以基于这些计算结果衍生，但不能替代原公式结果展示。

MVP 不追求：

- 一次性迁移全部复杂公式。
- 完整 CMS、权限、审批流。
- 经销商系统、报价系统、订单系统。
- 复杂 BI 报表。

## 3. 用户角色

### 3.1 工程师

特征：

- 熟悉设备工况与参数
- 需要准确计算与结果解释
- 关注安全余量、能量、行程、推力等约束

目标：

- 快速定位工况
- 输入参数后得到候选型号
- 明确为什么推荐这些型号

### 3.2 采购人员

特征：

- 不一定理解全部公式
- 更关心可买型号、规格差异、是否满足需求

目标：

- 尽量少填参数
- 快速筛出可用型号
- 能做简单对比与导出结果

---

## 4. 产品定义

### 4.1 产品定位

这是一个工业品在线选型工具，不是营销官网。

界面风格要求：

- 偏 B 端工具
- 信息密度适中
- 重点突出表单、计算结果、筛选表格
- 少装饰、多解释

### 4.2 使用方式

用户从首页进入后，不直接看到 19 个工况，而是通过上层分类与向导提问进入正确工况。

## 5. 工况分类设计

## 5.1 前台只暴露 5 个入口

### A. 直线运动 · 自由运动

适用于：

- 水平自由运动
- 斜坡自由滑动
- 垂直自由落体

### B. 直线运动 · 外力驱动

适用于：

- 水平附加推进力
- 垂直推进力（重力同向 / 反向）

### C. 直线运动 · 电机驱动

适用于：

- 水平电机驱动
- 垂直电机驱动（重力同向 / 反向）

### D. 直线运动 · 气缸驱动

适用于：

- 水平气缸驱动
- 垂直气缸驱动（重力同向 / 反向）

### E. 旋转运动

适用于：

- 水平旋转：工作台 / 横梁或挡板 / 负荷
- 垂直旋转：横梁或挡板 / 负荷 + 重力方向 / 重力反方向

## 5.2 向导式选择问题

为了让用户更容易选，工程师页先让用户回答以下问题：

1. 你的设备是直线运动还是旋转运动？
2. 安装方向是水平、垂直还是斜坡？
3. 运动是自由运动，还是由外力、电机、气缸驱动？
4. 如果是旋转，旋转对象是工作台、横梁/挡板，还是负荷？
5. 如果是垂直运动，运动方向与重力是同向还是反向？

系统根据回答自动匹配到底层工况 variant。

## 5.3 后台 calculator 建议分成 7 组

1. `linear_free_motion`
2. `linear_force_driven`
3. `linear_motor_driven`
4. `linear_cylinder_driven`
5. `rotary_load`
6. `rotary_beam_or_gate`
7. `rotary_table`

## 7. MVP 功能范围

## 7.1 必做

### 7.1.1 首页

功能：

- 展示工具用途
- 提供两个入口：工程师选型 / 采购快速筛选
- 提供分类说明

### 7.1.2 采购快速筛选页

用户无需走工况计算，直接按产品参数筛选。

筛选字段：

- 类型
- 最小缓冲行程
- 最小每次最大吸收能量
- 最小每小时最大吸收能量
- 最大冲击力下限
- 最大推进力下限
- 最大总长度上限
- 螺纹尺寸

输出：

- 符合条件的产品列表
- 支持排序
- 支持查看详情

### 7.1.3 工程师选型页

流程：

1. 选择上层工况分类
2. 回答向导式问题
3. 进入对应参数表单
4. 提交计算
5. 返回计算结果与匹配产品

计算结果展示要求：

- 结果区必须显示与 Excel 工况 sheet 一致的中间计算项，而不是只显示简化后的摘要卡片。
- 至少展示：动能、做功能量、总能量（每次）、总能量（每小时）、推进力。
- 如果对应工况 sheet 还定义了减速加速度、冲击速度、等效质量、重力修正等结果项，也必须一并展示。
- 工程师用户应能用页面结果直接与 Excel 样表逐项核对。

MVP 先实现 3 个示例工况：

- 水平-自由水平运动
- 水平-电机驱动
- 垂直-自由落体

### 7.1.4 产品库导入

- 读取 Excel 的“数据库”sheet
- 写入 products 表
- 保留原始字段快照 raw_data_json

### 7.1.5 工况注册表

- 每个工况需要有统一 key
- 记录所属 family、运动形式、方向、驱动方式、对象类型等元信息
- 存储 input schema 与输出 schema

### 7.1.6 计算接口

接口职责：

- 参数校验
- 单位换算
- 工况计算
- 生成筛选条件
- 查询产品库
- 返回推荐结果与解释

新增约束：

- 接口返回的 `calculation` 不能只保留筛选所需的压缩摘要值。
- 接口必须同时返回一个可直接渲染的“公式结果明细”结构，用于在前端结果区逐项展示原公式输出。

## 8. 非功能要求

### 8.1 可维护性

- 公式不能写在 React 页面组件里
- 表单、计算、数据库筛选必须分层
- 允许逐个工况迁移

### 8.2 可扩展性

- 新工况新增时，优先通过注册表 + calculator 实现
- 允许后续增加多语言、导出、分享链接

### 8.3 易用性

- 采购视图少术语、少公式
- 工程师视图给出必要解释
- 错误提示明确，字段单位统一

## 10. API 设计

### GET `/api/scenarios`

返回：

- 工况 family 列表
- family 下的引导问题与可选项
- 已启用的 scenario variants

### POST `/api/calculate`

请求：

- `familyKey`
- `variantKey`
- `input`

返回：

- 解析后的输入
- 计算结果
- 生成的筛选条件
- 匹配产品列表
- 推荐说明

### POST `/api/products/search`

请求：

- 采购筛选参数

返回：

- 产品列表
- 总数
- 排序结果

### POST `/api/import/excel`

开发期接口或脚本使用：

- 导入数据库 sheet
- 返回导入数量与异常记录

## 11. 数据要求

## 11.1 产品字段

来源于 Excel 数据库 sheet：

- 产品型号
- 类型
- 缓冲行程（mm）
- 每次最大吸收能量（Nm/c）
- 每小时最大吸收能量（Nm/h）
- 最大冲击力（N）
- 最大推进力（N）
- 总长度
- 螺纹尺寸
- 产品照片

## 11.2 工况字段

每个 scenario 至少包含：

- key
- name
- familyKey
- motionKind
- orientation
- driveType
- loadType
- gravityRelation
- inputSchemaJson
- outputSchemaJson
- formulaMetaJson

## 13. 验收标准

MVP 完成时，需要满足：

1. 项目可本地启动。
2. 数据库迁移可执行。
3. 能成功导入数据库 sheet。
4. 采购筛选页可正常筛选产品。
5. 工程师页能完成至少 3 个工况的计算与筛选。
6. 工况不以 19 个平铺按钮暴露给用户，而是用 5 个上层入口 + 向导式问题。
7. 代码结构清晰，新增工况不需要重写整页逻辑。
